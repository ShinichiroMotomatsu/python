# ころころスイッチこうぼう — 現行仕様書 (v6 / 全55面)

> 本書は現行実装 `korokoro.html` のリバース仕様。**AIに再実装・改修させるためのインプット**を想定し、
> 「要件 → データスキーマ → 挙動の数値 → 検証方法 → 過去にハマった落とし穴」の順で、実装判断に必要な情報をすべて明文化する。

---

## 1. コンセプトと対象ユーザー

- ピタゴラスイッチ風の物理パズル。**どうぐ(ギミック)をドラッグ配置 → 「ためす」→ ボールがゴールに入ればクリア**。
- 対象は **6歳児(未就学〜低学年)**。これが全UX判断の基準:
  - 文字は最小限・**すべて ひらがな/カタカナ**(漢字禁止)。
  - ボタンは大きく(半径26px以上)、ドラッグ&ドロップのみで完結。タイピングなし。
  - **失敗ペナルティなし**。失敗時はやさしいトースト(「おしい！ もういちど ためそう」)→自動で編集画面に戻る。
  - 詰まったら **ヒントボタンで3段階の助け舟**(後述)。答えを見てもクリア扱い。
- 遊びの核: 「置く→試す→直す」の高速ループ。1回の試行は数秒〜20秒程度。

## 2. 技術要件 (MUST)

| 項目 | 要件 |
|---|---|
| 配布形態 | **単一HTMLファイル**。外部リクエストゼロ(CDN不可・オフライン動作) |
| 物理エンジン | Matter.js **0.19.0** (MIT) を**ファイル内にインライン**。※CDNはプロキシ環境でブロックされるため、npm レジストリの tgz から取得して埋め込んだ経緯あり |
| 描画 | Canvas 2D。requestAnimationFrame の自前ループ(dt上限50ms) |
| 音 | WebAudio でオシレータ合成の簡易効果音(外部音源ファイルなし)。初回ポインタ操作で `AudioContext.resume()` |
| 保存 | `localStorage`(キー `korokoro_v1`)に `{stars: {レベル番号: true}}` |
| 入力 | Pointer Events(マウス/タッチ両対応、`setPointerCapture`使用)。マルチタッチは先着1本のみ有効 |
| 対応画面 | スマホ縦・スマホ横・PC。リサイズで動的切替 |

## 3. 座標系とレイアウト

- ステージ論理サイズ **800×480**(左上原点)。グリッド **48px**。
- **縦向き**(vw/vh ≦ 1.5): キャンバス論理 800×600。下120pxがどうぐ箱(横並びスロット)。
- **横向き**(vw/vh > 1.5): キャンバス論理 1008×480。右208pxがどうぐ箱(縦並びスロット、幅132×高90)。
- 表示は `min(vw/W, vh/H)` の等倍スケール + devicePixelRatio(上限2)。入力座標は同係数で論理座標へ逆変換。
- 主要UI: ⌂もどる(左上)・💡ヒント/↺もういちど(右上)・▶ためす(どうぐ箱側の大ボタン、置く物があるとき脈動)。

## 4. シーンフロー

```
select ──タップ──▶ edit ──▶ためす──▶ run ──win──▶ win(オーバーレイ) ──つぎへ──▶ 次のedit
                    ▲                    │fail(1.5s後 自動)              └もういちど─▶ 同じedit
                    └────────────────────┘   (配置は保持したまま戻る)
```

- **select**: 全レベルを丸ボタンでグリッド表示(縦5列/横7列、行間はレベル数に応じ自動圧縮)。クリア済み=金星、未開放=錠前。開放は「レベルnクリアでn+1開放」の直列。
- **edit**: 配置エリア(青い点線枠+グリッド点)内にどうぐを配置。ボールとゴールは常時表示。宙に浮いたボールには落下予告の点線矢印。
- **run**: 物理シミュレーション実行。↺で中断して edit へ(配置保持)。
- **win**: 紙吹雪+星3つ演出+「つぎへ/もういちど」。クリアフラグ保存。
- 初回プレイ(レベル1未クリア)かつ未配置のときだけ、どうぐ箱→配置エリアへの誘導矢印アニメを表示。

## 5. 入力仕様(ドラッグ&ドロップ)

1. **どうぐ箱から**: スロット押下(残数>0)でドラッグ開始。**タッチ時はどうぐを指の48px上に持ち上げて表示**(指で隠れない)。マウスは持ち上げなし。
2. **配置済みを再ドラッグ**: 本体付近(bbox+8px)押下で掴む。
3. **タップ(移動<10px かつ <400ms)**: そのどうぐの `flip` をトグル(flip対応品のみ)。効果音で反応。
4. **ドロップ**:
   - **判定はすべて「どうぐの表示位置」基準**(指の位置ではない)。※指基準にすると持ち上げ分ずれて地面に置けないバグになる(実際にあった)
   - どうぐ箱領域に重ねたら「もどす」(配置済みなら削除)。
   - それ以外は `findPlace`: グリッドにスナップ→置けなければ**周囲±3マスから最も近い置けるマスに吸着**。どこも不可なら効果音を鳴らして不発(配置済み品は元の位置のまま)。
5. **ドラッグ中プレビュー**: 吸着先に半透明のどうぐ+緑(可)/赤(不可)のハイライト枠を常時表示。

## 6. 配置ルール `canPlace(type, x, y)`

すべて満たすとき配置可:
- どうぐのbboxが**レベルの配置エリア(area)内**(誤差1px許容)。
- 画面内。`y ≦ 480`。
- **どのボールの開始位置からも** bboxまで `BALL_R+6` px 以上離れている(=ボールに触れる位置には置けない。※「ボールを乗せて発射」系の面は作れない制約)。
- ゴール矩形(60×60)と非重複。
- `switch`/`cannon` は**床の上のみ**(いずれかの static の上面と |top−y|<26 かつ x が幅+16px内)。
- ワープの穴(半径26+4)・ほし(半径16+6)に重ならない。
- static(地形)との重なり深さが許容以下: seesaw/pendulum=40px、mover/belt=24px、その他=14px(=「橋を架ける」系は床に少し食い込んでよい)。
- とびら(door)と非重複(2px)。
- 他の配置済み・fixed ギミックのbboxと非重複(3px)。※`fixedSwitch:true` の固定ボタンだけは重なり判定から除外

## 7. ギミック仕様(プレイヤーが置ける10種)

アンカーの規約: 特記なき場合 **(x,y)=下端中央**。`flip` は左右反転。

| type | 表示名 | サイズ(bbox) | flip | 挙動 |
|---|---|---|---|---|
| `ramp` | さか | 120×60 直角三角形 | ○ | 静的。flip=false「＼」(左が高い)。friction 0.002 |
| `spring` | ジャンプだい | 84×26 | − | 接触した瞬間 **vy = −9.5**(vxは接触直前の値を保持)。クールダウン350ms/ボール。縮み→伸びアニメ |
| `bumper` | はねかえし | 110×18 の板を±70°傾け(bbox 64×100) | ○ | 静的、restitution **1.4**(当たると強く跳ね返す)。flipで向き |
| `fan` | せんぷうき | 本体56×64 | ○ | 風ゾーン(前方230px・高さ〜110px)内のボールへ毎サブステップ **0.0009** の水平力 |
| `seesaw` | シーソー | 板160×16+台(bbox 168×78)。アンカー=板上面中心 | − | 通常: **動的な板を中心ピンで拘束**、±25°でクランプ。`level.seesawBridge:true` の面では**静的な水平板**(=橋)になる |
| `switch` | ふむボタン | 80×18 | − | ボールが上に**累計150ms**乗ると `switchColor` のトリガーをON(離れると0.5倍速で回復)。床上限定 |
| `mover` | うごくゆか | 板96×16、bbox は移動域込み 288×16。アンカー=中心 | − | x(t)=x₀+96·sin(2πt/2400ms)。**乗ったボールは板に剛体アタッチ**(載った相対位置を保持して setPosition/setVelocity 追従)。ゴール52px以内では解放(吸い込みを妨げない) |
| `pendulum` | ふりこ | 支点アンカー。ロッド90px+おもりr16、bbox=振れ域 | ○ | 角度=0.9·sin(2πt/2200ms)。おもりは**センサー**(衝突しない)。おもりが `dir` 方向へ動いている瞬間にボールが半径 16+18+4 内へ入ると **v=(dir·8.2, −6.0)** を設定(クールダウン260ms/ボール)。※「決まった向きに確実に飛ばす」が設計意図 |
| `cannon` | たいほう | 60×60。床上限定 | ○ | 口(x, y−22)の半径30内にボールが入ると口へ吸着後 **v=(dir·6.8, −9.6)** で発射(クールダウン500ms/ボール)。筒の描画は dir と一致させること(逆になるバグがあった) |
| `belt` | ベルト | 144×20。**アンカー=上面**(床と面一に置ける) | ○ | 静的な台+上のボールへ `vx += (dir·4.2 − vx)·0.3`/サブステップ(=一定速度へ収束) |

## 8. ステージ側のしかけ(レベルデータで固定配置)

- **statics**: 矩形の地形ブロック。`F(x1,x2)`=上面432の床、`B(x1,y1,x2,y2)`=任意矩形。上面に草の縁取りを描画。
- **トリガー連動品**(`triggers` + `switchColor`): スイッチと**同色**のものだけ作動。色は `blue`/`red`/`yellow`。
  - `Door(color,x,top,h)`: 幅26の縦扉。ON で上へスライドして消える(物理からも除去)。
  - `Bridge(color,x,top,w)`: ON で出現する横板(出るまで物理に存在しない)。
  - `TFan(color,x,y,dir)`: ON で回り出す扇風機。風力は通常fanと同じ+微小な上向き成分。
- **`Warp(ax,ay,bx,by)`**: 入口(青い渦・半径26)に入ると出口(オレンジの渦)へテレポート。**vxは±6でクランプして保持、vyは max(1.5, 0.3vy)**(=勢いを保って出る。ゼロにすると出口で停止して詰む)。クールダウン400ms/ボール。
- **`Star(x,y)`**: とると消える星(半径16)。**全部とるまでゴール不可**(ゴール判定・吸い込みの両方をゲート)。
- **複数ボール**: `balls:[{x,y},...]`(先頭=あか、2つ目=あお、3つ目=みどり)。`goalBall:n` で「ゴールしてよいボール」を限定(**助けるボールがゴールに入ってもクリアにならない**——これがないと協力面はすべて抜け道化する)。
- **`fixed`**: 最初から置いてある操作不可のどうぐ(通常品は半透明95%表示)。`fixedSwitch:true` は重なり判定除外の固定ボタン。

## 9. 実験(run)の物理と勝敗判定

- サブステップ **1000/120 ms**、描画フレームごとに `round(dt/substep)` 回実行。重力 y=1。
- ボール: 半径18、density 0.0008, friction 0.002, frictionStatic 0.5, frictionAir 0.0001, restitution 0.3。
- **ゴール**(旗+箱、60×60、`goal:{x,y}`=中心):
  - 中心46px以内で**吸い込み力** 0.00035(星未回収なら無効)。
  - 勝利 = ボール中心がゴール矩形内 **または** 接触300ms継続(goalBall・星ゲートを満たすとき)。
- **失敗**(いずれも run を止めて1.5秒後に自動で edit へ戻す。配置は保持):
  - `そとに でちゃった`: 全ボールが画面外(y>620 or x<−80 or x>880)。※goalBall 指定面では**助けボールの退場は失敗にしない**(aliveCount で管理)
  - `とまっちゃった`: runT>1200ms 以降、全生存ボールの速度<0.22 が1300ms継続。
  - `じかんぎれ`: `level.limit` 秒経過(面ごとに12〜26秒)。
- 演出: ボール軌跡(残像70点)、スプリング縮み、紙吹雪、トースト。

## 10. ヒントシステム(3段階) と レベルデータの二重役割

💡ボタンを押すたびに `hintStage` が1→2→3:
1. 文言ヒント表示(`hintText[0]`)+配置エリアが強く光る。
2. `hintText[1]`+**使うべきどうぐのスロットが金色に光る**。
3. `hintText[2]`+**答えの位置に半透明ゴースト表示**(同種を近く(±30px)に置いたものは消える)。

**重要**: `hints` 配列は「答えのゴースト」と「自動テストの解」を兼ねる。したがって
**hints の座標は必ず48の倍数**(プレイヤーはグリッドにしか置けないため。非整列だとテストは通るのに人間が再現できない/その逆が起きる)。

## 11. レベルデータ スキーマ

```js
{ name: 'ひらがなの なまえ',
  ball: {x,y},                    // または balls:[{x,y},...] + goalBall:n
  goal: {x,y},                    // ゴール中心
  statics: [F(0,800), B(...)],    // 地形
  area: R(x,y,w,h),               // 配置可能エリア(青枠)
  gadgets: {ramp:1, spring:2},    // 持ちどうぐと個数
  hints: [Hn('ramp',144,432,false), ...],  // ★答え。48の倍数必須
  hintText: ['...','...','...'],  // 3段階の文言(省略時は汎用文言)
  limit: 20,                      // 制限秒
  // 任意:
  fixed: [Hn(...), {type:'switch',x,y,flip:false,fixedSwitch:true}],
  triggers: [Door('blue',624,336,96), Bridge(...), TFan(...)],
  switchColor: 'blue',
  seesawBridge: true,             // この面のシーソーは静的な橋
  warps: [Warp(ax,ay,bx,by)],
  stars: [Star(x,y), ...],
}
```

## 12. 面構成(現行55面の設計意図)

| 帯 | 内容 |
|---|---|
| 1–10 | 基本4種(さか/ジャンプ/はねかえし/せんぷうき)を1つずつ→組み合わせ |
| 11–20 | シーソー(傾き/橋)・色つきスイッチ(とびら/はし/かぜ)の導入と応用 |
| 21–30 | 3段構成、**2ボール協力**(あかが道を作り あおがゴール)、ポケット+固定ボタン型 |
| 31–35 | うごくゆか・ふりこの導入と組み合わせ |
| 36–55 | **新システム帯**: たいほう/ベルト/ワープ/ほし集めの導入→合成(たいほう→ワープ→ほし 等) |

*子供からのフィードバック*: 既存ギミックの再組み合わせだけの追加面は「前と同じ」と見抜かれる。
**新しい動詞(飛ばす/運ぶ/瞬間移動/集める)を足すこと**が面の新鮮さに直結する。作り直しでもこの原則を推奨。

## 13. テスト容易性 (これが開発速度を決める。再実装でも必ず用意すること)

ページは `window.KOROKORO` を公開:

```js
KOROKORO.levels            // レベル数
KOROKORO.goto(i)           // i番へ(0-index)
KOROKORO.test(i)           // ヒント解でシミュレーション → {result:'win'|'fail', reason, t, trace}
KOROKORO.testAll()         // 全面一括
KOROKORO.state()           // {scene, levelIdx, runResult}
```

- `simulateSolution` は **hints をプレイヤーと同じくグリッドへスナップして canPlace を通す**(通らなければ error)。その後 `startRun(sim=true)` で音・演出を切って物理だけ回す。traceに0.5秒ごとの全ボール座標。
- ヘッドレス検証: Playwright(playwright-core + 同梱Chromium)で `file://` を開き `KOROKORO.testAll()`。判定基準:
  1. **全面 win**(ALL PASS)
  2. **0ギミック不成立**: hints を空にして win しないこと
  3. **1ギミック不成立**(持ちどうぐ2個以上の面): 全種×エリア内全グリッド×flip の総当たりで、どの1個置きでも win しないこと
- UI検証も Playwright の実ポインタ/タッチイベントで実施(ドラッグ配置、win遷移、ヒント巡回、失敗フロー)。

## 14. 落とし穴と設計判断の記録 (再実装時の必読事項)

1. **hints は48の倍数に**。スナップ後に canPlace を通らない座標は「テスト不能な答え」になる(§10)。
2. **動く静的ボディにボールを乗せない**。Matter.js で `Body.setPosition` する静的床は乗ったボールを弾き飛ばす。
   - シーソー → **動的な板+中心ピン拘束+角度クランプ**で解決。
   - うごくゆか → 板は静的でよいが、**ボール側を剛体アタッチ**(相対位置を保持して毎サブステップ setPosition)。摩擦や「そっと押す」方式は振幅が減衰して端まで運べない。
3. **ふりこは物理衝突させない**。おもりを実体にするとボールを弾いた向きが運次第になり、6歳向けの「たよれる道具」にならない。**センサー化+進行方向が合う瞬間だけ固定ベクトルで蹴る**。
4. **ゴール吸い込み(46px)は抜け道製造機**。落下線上にゴールがあると「何も置かずに落ちて吸われて勝ち」が発生する。0/1ギミック総当たりスキャンで機械的に検出する(§13)。
5. **たいほう等の描画向きと物理向きの一致確認**。flip の回転符号ミスは目視でしか気づけない(実際に発生)。ギミック追加時はスクリーンショット確認を手順に含める。
6. **canPlace はボール開始位置に触れる配置を禁止**している。よって「ボールの真下に発射台」のような面は成立しない。作りたければ仕様変更が必要。
7. **ワープ出口で速度をゼロにしない**(出口で停止して「とまっちゃった」になる)。
8. **判定は指でなくどうぐの表示位置**(§5)。タッチの持ち上げ(48px)分のズレに注意。
9. 協力面は必ず `goalBall` を付ける。付け忘れると助けボールが直接ゴールして崩壊する。
10. 新しい面は「実証済みの形のミラー/変形」より、**新しい動詞の導入**が子供の体感に効く(§12)。
11. ベルト等「床と面一」の道具はアンカーを上面にし、static との重なり許容(§6)を個別に広げる。
12. スイッチ踏み判定は接触でなく**領域内滞在150ms**(取りこぼし防止)。とびらは開いたら物理から除去、はしは出るまで物理に入れない。
