## 0. v7 での変更概要と動機

| # | 変更 | 動機(フィードバック) |
|---|---|---|
| 1 | **ふりこを物理挙動に**: おもり部分だけ実体の当たり判定を持ち、重力で振れる。スクリプトによる固定ベクトルの蹴り出しを廃止 | 「見た目と挙動が一致し、物理法則に沿って動いてほしい」 |
| 2 | **うごくゆかを物理挙動に**: 板が実体で、上のボールは摩擦で運ばれる。剛体アタッチ(吸着)を廃止 | 同上 |

## 1. コンセプトと対象ユーザー

- ピタゴラスイッチ風の物理パズル。**どうぐ(ギミック)をドラッグ配置 → 「ためす」→ ボールがゴールに入ればクリア**。
- 対象は **6歳児(未就学〜低学年)**。これが全UX判断の基準:
  - 文字は最小限・**すべて ひらがな/カタカナ**(漢字禁止)。
  - ボタンは大きく(半径26px以上)、ドラッグ&ドロップのみで完結。タイピングなし。
  - **失敗ペナルティなし**。失敗時はやさしいトースト(「おしい！ もういちど ためそう」)→自動で編集画面に戻る(配置保持)。
  - 詰まったら **ヒントボタンで3段階の助け舟**(§10)。答えを見てもクリア扱い。
- 遊びの核: 「置く→試す→直す」の高速ループ。1回の試行は数秒〜30秒程度。
- **(v7変更)** 難易度の軸は「操作の難しさ」ではなく**因果の連鎖を組み立てる思考**に置く。個々のどうぐは単体では必ず「たよれる」(同じ置き方なら毎回同じ結果)こと。

## 3. 座標系とレイアウト (v7変更: グリッドとスケール)

- ステージ論理サイズ **800×480**(左上原点)。グリッド **32px**(旧48)。→ 配置解像度 25×15 マス。
- ボール半径 **12**(旧18)。ギミック寸法は旧比 約2/3(§7)。
- **縦向き**(vw/vh ≦ 1.5): キャンバス論理 800×600。下120pxがどうぐ箱(横並びスロット)。
  - **(v7変更)** どうぐ種類が多い面(5種以上)でスロットが収まらない場合、どうぐ箱は横スクロール(スワイプ)可能にする。スロット幅は最小88pxまで縮めてよい。
- **横向き**(vw/vh > 1.5): キャンバス論理 1008×480。右208pxがどうぐ箱(縦並び、必要なら縦スクロール)。
- 表示は `min(vw/W, vh/H)` の等倍スケール + devicePixelRatio(上限2)。入力座標は同係数で論理座標へ逆変換。
- 主要UI: ⌂もどる(左上)・💡ヒント/↺もういちど(右上)・▶ためす(どうぐ箱側の大ボタン、置く物があるとき脈動)。

## 4. シーンフロー (不変)

```
select ──タップ──▶ edit ──▶ためす──▶ run ──win──▶ win(オーバーレイ) ──つぎへ──▶ 次のedit
                    ▲                    │fail(1.5s後 自動)              └もういちど─▶ 同じedit
                    └────────────────────┘   (配置は保持したまま戻る)
```

- **select**: 全レベルを丸ボタンでグリッド表示(縦5列/横7列、行間自動圧縮)。クリア済み=金星、未開放=錠前。開放は直列(レベルnクリアでn+1開放)。
- **edit**: 配置エリア(青い点線枠+グリッド点)内にどうぐを配置。宙に浮いたボールには落下予告の点線矢印。
- **run**: 物理シミュレーション実行。↺で中断して edit へ(配置保持)。
- **win**: 紙吹雪+星演出+「つぎへ/もういちど」。クリアフラグ保存。
- 初回プレイ(レベル1未クリア)かつ未配置のときだけ、どうぐ箱→配置エリアへの誘導矢印アニメ。

## 5. 入力仕様(ドラッグ&ドロップ) (不変・数値のみ更新)

1. **どうぐ箱から**: スロット押下(残数>0)でドラッグ開始。**タッチ時はどうぐを指の48px上に持ち上げて表示**(指で隠れない)。マウスは持ち上げなし。
2. **配置済みを再ドラッグ**: 本体付近(bbox+8px)押下で掴む。
3. **タップ(移動<10px かつ <400ms)**: `flip` をトグル(flip対応品のみ)。効果音で反応。
4. **ドロップ**:
   - **判定はすべて「どうぐの表示位置」基準**(指の位置ではない)。※指基準だと持ち上げ分ずれて地面に置けないバグになる(v6で実際に発生)
   - どうぐ箱領域に重ねたら「もどす」(配置済みなら削除)。
   - それ以外は `findPlace`: グリッドにスナップ→置けなければ**周囲±3マス(=±96px)から最も近い置けるマスに吸着**。どこも不可なら効果音を鳴らして不発。
5. **ドラッグ中プレビュー**: 吸着先に半透明のどうぐ+緑(可)/赤(不可)のハイライト枠を常時表示。

## 6. 配置ルール `canPlace(type, x, y)` (数値をv7スケールに更新)

すべて満たすとき配置可:
- どうぐのbboxが**レベルの配置エリア(area)内**(誤差1px許容)。
- 画面内。`y ≦ 480`。
- **どのボールの開始位置からも** bboxまで `BALL_R+4` px 以上離れている。
- ゴール矩形(44×44 ※要調整)と非重複。
- `switch_*`/`cannon`/`domino` は**床の上のみ**(いずれかの static の上面と |top−y|<18 かつ x が幅+12px内)。
- `domino` 同士は16px(半マス)以上離す(§7.3)。
- ワープの穴(半径18+4)・ほし(半径12+4)に重ならない。
- static(地形)との重なり深さ許容: seesaw/pendulum=28px、mover/belt=16px、その他=10px(=「橋を架ける」系は床に少し食い込んでよい)。
- とびら(door)と非重複(2px)。
- 他の配置済み・fixed ギミックのbboxと非重複(2px)。※`fixedSwitch:true` の固定ボタンだけは除外

### 7.1 うごくゆか(物理版)の要求仕様 (v7変更)

- 板は**実体の当たり判定**を持つ。x(t) = x₀ + 64·sin(2πt/2400ms) で往復(※振幅・周期は要調整)。
- **ボールはスクリプトで吸着させない**。運搬は板とボールの**接触摩擦**で行う:
  - 板上面の friction は高く(0.8〜1.0)、ボール側は現状(0.002)のまま。
  - 実装指針: 毎サブステップ、板を `Body.setPosition` で軌道に沿って動かすと**同時に** `Body.setVelocity` で板の真の速度(dx/dt)を設定する。Matter の衝突解決は body の velocity を使うため、位置だけ動かす(テレポート)と摩擦が働かずボールが置き去り/弾き飛ばしになる(v6で実際に発生した不具合。§14-2)。
- 結果として満たすべき観測可能な性質(受け入れテスト。§13.3):
  1. 板の中央に落としたボールが、**3往復のあいだ板から落ちずに運ばれる**(端に置いた場合に自然に転がり落ちるのは正しい挙動)。
  2. 板が横からボールに当たったとき、ボールは**押される**(すり抜け・埋まり・上方への弾き出しが起きない)。
  3. 同じ配置なら毎回同じ結果(決定性)。
- 折り返し点では板の速度が0になるため、慣性でボールが板上を少し滑るのは**許容**(物理的に正しい)。滑り落ちやすさは板の摩擦と振幅・周期で調整する。

### 7.2 ふりこ(物理版)の要求仕様 (v7変更)

- **当たり判定は「おもり」円(r12)のみ**。ロッド・支点は非衝突(描画のみ)。
- おもりは**動的ボディ**とし、支点へ**剛体拘束**(Constraint: stiffness 1, length=ロッド長64)でぶら下げる。シーソーで実績のある「動的ボディ+ピン拘束」方式(§14-2)。
- **重力で振れる**。初期状態は flip の側へ振り上げた角度(±0.9rad ※要調整)から静かに放す → 以後は物理任せ。
- 減衰対策: frictionAir はごく小さく(≦0.001)。それでも実用上減衰が目立つ場合のみ、**振幅維持の最小限のエネルギー補給**(例: 最下点通過時に角速度を目標値へ僅かに近づける)を許容する。ただし
  - 補給はボールとの衝突応答そのものには介入しないこと(蹴り出しベクトルの直接設定は禁止)。
- おもりの質量はボールより十分重く(密度 ≥ 5×ボール ※要調整): ボールを叩けば飛ばせるが、ボールが当たってもおもりの振れが大きくは乱れない、を両立する。
- 結果として満たすべき観測可能な性質(受け入れテスト。§13.3):
  1. おもりは制限時間(〜40s)のあいだ振れ続ける(振幅の目減りが体感で分からない程度)。
  2. 最下点付近で順方向に振れているおもりにボールが触れると、ボールは**振れの向きへ水平に飛ばされる**(獲得速度はおおむね一定=たよれる)。
  3. 逆方向に振れている瞬間に当たるとボールは押し戻される(物理的に正しい。**面設計側でタイミング依存になりすぎないよう配置でケアする**: 例=ボールを溜める窪みを設け、順方向スイングで確実に掬われる形にする)。
  4. ロッド部分をボールが素通りできる。
  5. 同じ配置なら毎回同じ結果(決定性)。
- **設計上の含意**: v6の「センサー+固定ベクトル」は6歳向けの確実性を優先した妥協だった。v7は物理化するぶん、**確実性は面ジオメトリ側で担保**する(受け皿・壁・溜まり場)。テスト(§13)が通らない配置は面として採用しない。

### 7.3 ドミノの要求仕様 (v7新規・物理)

- 形状: **10×48 の縦長の動的ボディ**(角丸小)。アンカー=下端中央。床上限定(§6)。flipなし(左右対称、倒れる向きは当たられた方向で決まる)。
- **完全に物理挙動**: スクリプトによる倒れ判定・連鎖判定は書かない。倒れる/倒れないは Matter.js の剛体シミュレーションのみで決まる。
  - 密度はボールと同程度〜やや軽め(※要調整): 転がってきたボールが1枚目を倒せる/倒れた1枚が次の1枚を倒せる、の両立点を§13.3のハーネスで探す。
  - 床との摩擦は高め(0.8〜 ※要調整): 押されたとき滑らずに**下端を支点に回転して倒れる**こと。restitution は低く(≦0.1、跳ねて暴れない)。
  - 空気抵抗は最小(≦0.001)。
- **連鎖の成立条件をプレイヤーの配置課題にする**: 隣のドミノとの間隔が高さ(48px)より狭ければ倒れた板が次に届く。グリッド32px間隔で並べれば確実に連鎖し(48>32)、2マス(64px)空けると届かない——「**1マスおきに並べる**」が子供に発見させたい正解。
- 用途(面設計の道具立て):
  1. **信号の水平伝達**: ボールがわずかに触れるだけで、離れた場所まで「倒れ」が伝わる。
  2. **スイッチ押し**: 倒れたドミノがボタン上に載れば作動する(§7のswitch仕様)。=「ボールが直接踏めない場所のスイッチをドミノ連鎖で押す」という多手数の核になる。
  3. 倒れたドミノは倒れたまま(起き上がらない)。↺リセットで復元。
- canPlace 追加規則: ドミノ同士は**16px(半マス)以上**離す(重ね置きで初手から倒れるのを防止)。置いた直後(edit中)は静止して立っていること(run開始まで物理を回さない、は従来どおり)。
- 受け入れテスト(§13.3):
  1. 32px間隔で5枚並べ、ボールを転がして1枚目に当てる → **5枚すべてが倒れる**(所要≦4秒)。
  2. 64px間隔では2枚目以降が**倒れない**。
  3. 5枚目の先に置いたスイッチが、倒れた5枚目により**ONになる**。
  4. 同じ配置なら毎回同じ結果(決定性)。
- 停止判定との関係: **ドミノが動いている(速度・角速度が閾値超)あいだは全ボールの停止タイマーをリセット**する(§9)。ボールが役目を終えて止まっても、連鎖が進行中なら失敗にしない。

## 8. ステージ側のしかけ(レベルデータで固定配置)

- **statics**: 矩形の地形ブロック。`F(x1,x2)`=上面432の床、`B(x1,y1,x2,y2)`=任意矩形。上面に草の縁取り。
- **トリガー連動品**(`triggers`): **(v7変更)** 1面に**複数色**を共存可能。作動条件は「同色の switch_* が押されること」。
  - `Door(color,x,top,h)`: 幅18の縦扉。ON で上へスライドして消える(物理からも除去)。
  - `Bridge(color,x,top,w)`: ON で出現する横板(出るまで物理に存在しない)。
  - `TFan(color,x,y,dir)`: ON で回り出す扇風機。
  - **(v7新規・任意実装)** `Lift(color,x,y,dy)`: ON で dy だけ上下する床。多手数面の道具立てとして有用なら追加してよい(§7.1と同じ「setPosition+setVelocity 併用」で実装)。
- **`Warp(ax,ay,bx,by)`**: 入口(青い渦・半径18)→出口(オレンジ)へテレポート。**vxは±6でクランプして保持、vyは max(1.5, 0.3vy)**(勢いを保つ。ゼロにすると出口で停止して詰む)。クールダウン400ms/ボール。
- **`Star(x,y)`**: とると消える星(半径12)。**全部とるまでゴール不可**(ゴール判定・吸い込み両方をゲート)。
- **複数ボール**: `balls:[{x,y},...]`(あか/あお/みどり)。`goalBall:n` で「ゴールしてよいボール」を限定(**協力面に必須**。付け忘れると助けボールが直接ゴールして崩壊する)。
- **`fixed`**: 最初から置いてある操作不可のどうぐ(半透明95%表示)。`fixedSwitch:true` は重なり判定除外の固定ボタン。

## 9. 実験(run)の物理と勝敗判定 (v7変更: 停止判定の緩和)

- サブステップ **1000/120 ms**。重力 y=1。
- ボール: 半径12、density 0.0008, friction 0.002, frictionStatic 0.5, frictionAir 0.0001, restitution 0.3(※要調整)。
- **ゴール**(旗+箱、44×44、`goal:{x,y}`=中心):
  - 中心34px以内で**吸い込み力** 0.00025(星未回収なら無効)(※要調整)。
  - 勝利 = ボール中心がゴール矩形内 **または** 接触300ms継続(goalBall・星ゲートを満たすとき)。
- **失敗**(いずれも run を止めて1.5秒後に自動で edit へ。配置保持):
  - `そとに でちゃった`: 全ボールが画面外(y>620 or x<−80 or x>880)。goalBall 指定面では助けボールの退場は失敗にしない。
  - `とまっちゃった` **(v7変更)**: runT>1200ms 以降、全生存ボールの速度<0.22 が **2500ms** 継続(旧1300ms)。さらに**トリガーON・ほし取得・ワープ・たいほう発射が起きるたび、およびドミノが動いているあいだ**、全ボールの停止タイマーをリセットする。※多手数面では「片方のボールが待つ」「ボールは止まったが連鎖が進行中」が正当な状態のため。
  - `じかんぎれ`: `level.limit` 秒経過。**(v7変更)** 上限を40秒まで許容(多手数面向け。標準は12〜30)。
- 演出: ボール軌跡(残像)、スプリング縮み、紙吹雪、トースト(「おしい！ もういちど ためそう」等2種ローテーション)。

## 10. ヒントシステム(3段階) と レベルデータの二重役割 (不変)

💡ボタンを押すたびに `hintStage` 1→2→3:
1. 文言ヒント(`hintText[0]`)+配置エリアが強く光る。
2. `hintText[1]`+**使うべきどうぐのスロットが金色に光る**。
3. `hintText[2]`+**答えの位置に半透明ゴースト表示**(同種を近く(±20px)に置いた分は消える)。

**重要**: `hints` 配列は「答えのゴースト」と「自動テストの解」を兼ねる。
**(v7変更)** hints の座標は必ず **32の倍数**(グリッドに一致。非整列だと「テストは通るのに人間が再現できない」事故になる)。

## 11. レベルデータ スキーマ (v7変更: スイッチ色分離を反映)

```js
{ name: 'ひらがなの なまえ',
  ball: {x,y},                    // または balls:[{x,y},...] + goalBall:n
  goal: {x,y},                    // ゴール中心
  statics: [F(0,800), B(...)],    // 地形
  area: R(x,y,w,h),               // 配置可能エリア(青枠)。広い面ではステージほぼ全域でよい
  gadgets: {ramp:2, cannon:1, switch_red:1, domino:5},   // 持ちどうぐと個数(スイッチは色別)
  hints: [Hn('ramp',128,432,false), ...],      // ★答え。32の倍数必須
  hintText: ['...','...','...'],  // 3段階の文言
  limit: 30,                      // 制限秒(≦40)
  // 任意:
  fixed: [Hn(...), {type:'switch_blue',x,y,flip:false,fixedSwitch:true}],
  triggers: [Door('blue',624,336,96), Bridge('red',...), TFan('yellow',...)],  // 複数色可
  seesawBridge: true,             // この面のシーソーは静的な橋
  warps: [Warp(ax,ay,bx,by)],
  stars: [Star(x,y), ...],
  branch: true,                   // 分岐面(§12の型)。§13.2-5 のスキャン対象。'timing' ならタイミング分岐(§13.2-6も実施)
}
```

(旧 `switchColor` フィールドは廃止。トリガーの色は各 trigger 自身が持ち、対応する色の switch_* で作動する)

## 12. 面設計の原則 (v7変更: 多手数化)

**ステップ数**の定義: 意図解における「因果リンク」の数。
例:「さかで転がす→たいほうに入る→発射されてスイッチ台に届く→あかボタンON→あかのはし出現→あおボールが渡る→ゴール」= 5ステップ。

| 帯 | 面数目安 | ステップ数 | 内容 |
|---|---|---|---|
| 導入 | 1–10 | 1–2 | 基本4種(さか/ジャンプ/はねかえし/せんぷうき)を1つずつ→2個組み合わせ |
| 中盤A | 11–20 | 2–3 | シーソー・色スイッチ(とびら/はし/かぜ)・ほし・ワープ・**ドミノ**・**分岐地形**の導入 |
| 中盤B | 21–35 | 3–4 | 物理ふりこ・うごくゆか・たいほう・ベルトの導入と3個以上の組み合わせ、2ボール協力の基本形 |
| 後半 | 36–55 | **4–6** | **多手数必須帯**(下記要件) |

**(v7新規) 分岐面の型**: 地形が2つ以上のルートに分かれ、**正しいルートへ導かないとクリアできない**面。
- 分岐の選択は、置くどうぐの**位置や向き(flip)**で決まるようにする(例: 分かれ道の頭上に「さか」を置き、＼なら右ルート・∕なら左ルートへ落ちる。はねかえし・せんぷうき・ベルトの向きでも可)。
- **誤ルートは必ず行き止まり**にする: 落下して退場(そとに でちゃった)、または袋小路で停止(とまっちゃった)して、やさしく失敗→再挑戦に戻ること。誤ルートからゴールへ回り込める地形にしない(§13.2-5で機械検証)。
- 正誤のフィードバックを地形で見せる: 誤ルートの先に何もない/正ルートの先にゴールや星が見える、など**見て比べられる**こと(6歳が「あっちだったか」と自力で気づける)。
- 分岐に星を組み合わせてよい(例: 星が誤ルート側にあり、まず星ルート→戻れないので2回目の仕掛けで本ルート、はv7では不可。**1回の実験で完結**する範囲にすること。星は正ルート上に置く)。
- 本数要件: 中盤A以降で**計6面以上**(うち後半帯に2面以上、2ボール協力との複合を1面以上)。

**(v7新規) タイミング分岐の型**(分岐面の発展形):
- 分岐点に**周期運動する仕掛け**(うごくゆか・ふりこ・トリガー連動リフト等)を置き、**ボールが到着した瞬間の状態**でルートが決まる。例:
  - うごくゆかが2つの穴の上を往復しており、到着時に板が左の穴を塞いでいれば板に乗って右ルートへ運ばれ、開いていれば下ルートへ落ちる。
  - ふりこが順方向に振れている瞬間に着けば打ち返されて上ルートへ、戻りの瞬間なら足元を通過して下ルートへ。
- **リアルタイム操作は導入しない**(実験中は見守るだけ、の原則を維持)。プレイヤーがタイミングを制御する手段は**配置**である: 遠回りの地形を通す・ベルトで加減速する・ワープを経由する等で**到着時刻そのものを設計する**。
- 決定性の担保: 周期運動の位相は**実験開始時に毎回リセット**(t=0から同じ動き)。したがって同じ配置なら毎回同じルートに入る(§2の決定性と両立)。「じっけんの たびに おなじ うごきで はじまる」こと自体を子供が学び、安心して試行錯誤できる。
- edit中のプレビューアニメは「動く仕掛けである」ことの示唆であり、実験開始時の位相と一致しなくてよい(一致は原理的に不可能)。
- **剃刀の刃の解は不採用**: 意図解の到着時間窓は十分広くとる(目安≧400ms)。検証は§13.2-6。周期は遅め(2秒以上)にし、窓を体感できる速度にする。
- 本数要件: **2面以上**(後半帯)。上記6面の分岐面と重複してよい。

**後半帯(36–55)の必須要件**:
- 全面で持ちどうぐ **3個以上**(推奨4–6個)。
- **5面以上**: 2色以上のトリガーが共存(例: あかのはしを出してから あおのとびらを開ける、順序依存)。
- **5面以上**: 2ボール協力かつ**助けるボール自身が2個以上のギミックを要する**(例: あかボールは さか+たいほう を置かないとスイッチに届かない)。
- **3面以上**: 3段以上の地形。
- **3面以上**: ドミノ連鎖が解の一部(うち1面以上は「ドミノがスイッチを押す」)。
- **2面以上**: 分岐地形(§12の型)。
- 露骨な一本道にしない: 配置エリアを広く取り(盤面の過半)、「どこに何を置くか」の探索自体が遊びになるようにする。
- ヒント3(ゴースト)は全どうぐ分を表示する(多手数でも詰み回避はヒントで保証)。

**新鮮さの原則(継承)**: 同帯内で「同じ動詞の並べ替え」だけの面を続けない。動詞(転がす/跳ぶ/押す/飛ばす/運ぶ/瞬間移動/集める/待つ/**倒す**/**えらぶ(分岐)**)の組み合わせが前の面と変わるように並べる。

## 13. テスト容易性 (開発速度を決める。最初に用意すること)

### 13.1 ページ公開API

```js
KOROKORO.levels            // レベル数
KOROKORO.goto(i)           // i番へ(0-index)
KOROKORO.test(i)           // ヒント解でシミュレーション → {result:'win'|'fail', reason, t, trace}
KOROKORO.testAll()         // 全面一括
KOROKORO.state()           // {scene, levelIdx, runResult}
```

- `simulateSolution` は **hints をプレイヤーと同じくグリッドへスナップして canPlace を通す**(通らなければ error)。音・演出を切って物理だけ回す。trace に0.5秒ごとの全ボール座標。
- 物理が固定サブステップである限り、シミュレーションは決定的(§2)。

### 13.2 面の合格基準(ヘッドレス Playwright + file:// で自動実行)

1. **全面 win**(ALL PASS)。
2. **0ギミック不成立**: hints を空にして win しないこと。
3. **1ギミック不成立**(持ちどうぐ2個以上の面): 全種×エリア内全グリッド×flip の総当たりで、どの1個置きでも win しないこと。
4. **(v7新規) 全部必要スキャン**: hints から**どの1個を除いても** win しないこと(=n回のシミュレーション。多手数面で「実は3個で解ける」を防ぎ、ステップ数の実効性を保証する)。
5. **(v7新規) 分岐チェック**(分岐面のみ): hints のうち**ルート選択を担うどうぐの flip を反転**(または誤ルート側の対称位置へ移動)した変種で win **しない**こと(=誤ルートが本当に行き止まりであることの機械検証)。分岐面にはレベルデータに `branch: true` を付けてスキャン対象を明示する。
6. **(v7新規) タイミング窓チェック**(タイミング分岐面のみ、`branch: 'timing'`): 到着時刻を決めるどうぐ(経路上の さか・ベルト等)を**±1グリッドずらした変種でも win** すること(=1マスのずれで壊れる剃刀の刃の解を不採用にする。§12)。

### 13.3 (v7新規) ギミック単体ハーネス

面テストとは別に、物理化したギミックの性質を固定の試験場で検証する:
- **mover**: 板中央に落としたボールが3往復落ちない/横から当たるとボールが押される/めり込み・上方弾きなし。
- **pendulum**: 40秒振れ続ける/最下点順方向ヒットでボールが規定距離(±20%)飛ぶ/ロッドは素通り/おもり実体。
- **domino**: 32px間隔×5枚が全倒れ(≦4秒)/64px間隔では連鎖しない/倒れた5枚目がスイッチを押せる(§7.3)。
- **spring**: 跳躍頂点が台の上面から**90〜100px**(約3グリッド)に収まること(強すぎ検知。§7)。
- **cannon・belt・warp**: 発射速度・搬送速度・テレポート後速度が仕様値どおり。
- これらをリグレッションとして `testAll` と同じスクリプトで常時実行する。

### 13.4 UI検証

Playwright の実ポインタ/タッチイベントで: ドラッグ配置(壁ぎわ吸着含む)・タップ反転・どうぐ箱へ戻す・win遷移・失敗フロー・ヒント3段階・縦横レイアウト・(v7)どうぐ箱スクロール。
**ギミック追加/変更時はスクリーンショット目視確認を必須手順とする**(向き・重なりの不整合はテストでは捕まらない。§14-5)。

## 14. 落とし穴と設計判断の記録 (再実装時の必読事項)

1. **hints はグリッドの倍数に**(v7では32)。スナップ後に canPlace を通らない座標は「テスト不能な答え」になる。
2. **動く床の実装**: 静的ボディを `setPosition` だけで動かすと、乗ったボールを弾き飛ばす/置き去りにする(v6で発生)。v7は **setPosition+setVelocity の併用**で衝突解決に正しい速度を渡し、摩擦で運ぶ(§7.1)。回転系(シーソー・ふりこ)は**動的ボディ+ピン拘束**が実績あり。
3. **ふりこ**: v6はおもりを非衝突センサー+固定ベクトル蹴り出しにしていた(確実性優先)。v7は物理化(§7.2)。その代償の不確実性(逆スイングに当たる等)は**面ジオメトリで受ける**こと。タイミング運ゲーになった面は不採用。
4. **ゴール吸い込みは抜け道製造機**。落下線上にゴールがあると「何も置かずに勝ち」が発生する。§13.2のスキャンで機械的に検出する。
5. **描画向きと物理向きの一致確認**(たいほうの筒が逆を向くバグがv6で発生)。スクリーンショット確認を手順に含める。
6. **canPlace はボール開始位置に触れる配置を禁止**。「ボールの真下に発射台」のような面は成立しない(作りたければ本ルールの仕様変更が先)。
7. **ワープ出口で速度をゼロにしない**(出口で停止して「とまっちゃった」になる)。
8. **ドロップ判定は指でなくどうぐの表示位置**(タッチの持ち上げ48px分のズレ)。
9. **協力面は必ず `goalBall`**。
10. **多手数面の停止判定**: 待機ボールを失敗扱いしない(§9の緩和+イベントでのタイマーリセット)。長い面では limit も忘れず延ばす。
11. 床と面一の道具(belt等)はアンカーを上面にし、static重なり許容を個別に広げる(§6)。
12. スイッチは接触でなく**領域内滞在150ms**(取りこぼし防止)。とびらは開いたら物理から除去、はしは出るまで物理に入れない。
13. **(v7新規) 多手数面は「全部必要スキャン」(§13.2-4)を必ず通す**。ステップを増やしたつもりでも物理が近道を見つけることが多い(v6でスプリング1個が2ギミック面を解いた例あり)。**ジャンプだいは特に要注意**: v6では跳躍が強すぎて壁・段差を意図せず越え、easy clear の主犯だった。v7で弱体化(§7)したが、スキャンでの検証は継続する。
14. **(v7新規) 物理ふりこ/うごくゆか/ドミノの調整はハーネス(§13.3)を先に作ってから**。面を作りながらパラメータをいじると、既存面が音もなく壊れる。
15. **(v7新規) ドミノの連鎖はパラメータに敏感**(密度・床摩擦・間隔)。ハーネスの「32pxで連鎖する/64pxで連鎖しない」の両側を守ること。片側だけ見て調整すると「どこに置いても倒れる(考える必要がない)」か「正解の間隔でも倒れない(理不尽)」になる。倒れたドミノが床で跳ねて暴れないよう restitution は低く保つ。
16. **(v7新規) 分岐面は誤ルートの「詰み方」もテストする**(§13.2-5)。誤ルートから偶然ゴールに転がり込める地形は分岐の意味を失う。
17. **(v7新規) タイミング分岐は「配置で時刻を作る」遊びであり、反射神経の遊びにしない**。周期は2秒以上・到着窓は400ms以上(§12)。位相は実験開始で毎回リセットし、±1グリッドずらしチェック(§13.2-6)で剃刀の刃の解を排除する。

## 15. 既知の制約・スコープ外

- 星(クリア評価)は演出のみ。性能評価(使用個数/時間)はしない。
- BGMなし(効果音のみ)。設定画面・音量調整・データリセットUIなし。
- undo/redoなし(↺は「全消し or 実験中断」)。
- レベルは直列アンロックのみ。スキップ不可。
- 多言語なし(日本語のみ)。
- v6セーブ(`korokoro_v1`)からの引き継ぎはしない(面が別物のため)。

## 16. (v7新規) チューニング手順

※要調整の定数(ボール物性・発射/搬送速度・ふりこ周期/質量・吸い込み半径 等)は次の順で確定する:
1. §13.3のギミック単体ハーネスを先に実装し、各性質が満たされるまで定数を調整。
2. 導入帯の面(1どうぐ)で「6歳が3試行以内に置ける」感触をスクリーンショット+実機で確認。
3. 以降、面を追加するたびに `testAll` + §13.2 スキャンを全面に回す(定数変更が過去の面を壊していないか常時検知)。
4. 定数を変更したら本仕様書の値も更新する(仕様書と実装の乖離を残さない)。

### 16.1 確定した定数 (korokoro2.html 実装で検証済み)

- ボール: `Bodies.polygon(30角形, r=12)` を使うこと。`Bodies.circle` は r12 だと12角形に離散化され転がりが極端に悪くなる(v7実装最大の落とし穴)。物性: density 0.0008 / friction 0.0005 / frictionStatic 0.05 / frictionAir 0.0001 / restitution 0.15。`circleRadius` を手動で設定。
- `Matter.Resolver._restingThresh = 1.6`(既定4のままだと v7 の低速域で反発が全て殺される)。
- 静的地形: friction 0.04。平地の転がり減衰は非常に小さい(400px走って vx −0.2程度)。
- ジャンプ台: 射出 vy=−7.2 → 頂点 ≒ 92px(約3グリッド)。
- たいほう: 捕捉半径20(x, y−15中心)、射出 (dir·5.5, −8.0) → 頂点101px・平地射程345px。
- ベルト: 上面アンカー96×14、目標 vx 3.4。
- ふりこ(物理振り子): 動的26角形ボブ r12(density 0.004, frictionAir 0.0005, restitution 0.4)+ピン Constraint(stiffness 1, length 80)。脱進機: |角度|<0.3 で速度を min(1.06, 4.3·0.98/sp) 倍、停止時 0.3 を注入 → 振幅 0.96rad を無期限維持。バット初速 ~5。設計アトム: 支点は床上面+96(32倍数)、真下に幅32・深さ4px のくぼみ、ボールは上から落として溜め、ボブが水平に打ち出す。打つ向きはボール到着時のボブの通過方向で決まる(flip は初期位相の反転。到着タイミングで逆に打たれる場合は flip を切り替えて合わせる。くぼみ周辺 ±80px にボブの弧を遮る壁を置かないこと)。
- うごくゆか: 静的プランク64×14を毎サブステップ `Body.setPosition(body, pos, true)` で移動。摩擦アシスト: プランク上面近傍(topY−18..−8, cx±48)のボールに vx を `dx·2`(=dx·(1000/60)/RUN_SUBSTEP)へ50%ブレンド。ゴール吸い込み圏+18px 内ではアシストを切る(リリース)。`moverPhase` が面ごとの時刻設計ノブ(タイミング分岐面は 0.625π を使用)。
- ドミノ: 10×48 動的矩形(density 0.0004, friction 0.3, frictionStatic 0.5, restitution 0.05, chamfer 2)。ジャム対策: 0.12<|角|<1.45 かつ |角速度|<0.015 → 角速度 sign·0.06 を注入。32px(1マス)間隔で5/5連鎖(~3.3s)、64pxでは連鎖しない。倒れた板は |角|>0.5 かつ スイッチ±48px でボタンを押せる。
- スイッチ: 滞在100ms・ゾーン x±32 / y−36..+12(ドミノは±48)。とびら開放時、とびら±44px のボールへ vx=1.6 のナッジ(とびら前待機のデッドエンド解消)。ナッジは無回転で滑るため到達距離は約70〜120px — ナッジ頼みの面はゴールをその範囲に置く。
- はねかえし: collisionStart で明示的に鏡面反射(法線は本体角度から、速度 max(3.4, sp·1.1))。restitution は低速では発火しない。※地面すれすれの進入は flip に関係なく手前に跳ね返る — バンパー分岐は「落下中のボール」にのみ有効。
- ゴール: GOAL_SUCK=34(force 0.00025)。閉じたとびら越しに吸い込まれない距離(とびら面+ボール半径から34px超)にゴールを置くこと。

### 16.2 仕様からの逸脱(許容済み)

- ドミノ連鎖の「1枚目」は §13.2 の全必要性スキャンで DISPENSABLE になり得る(ボール自身の転がりが連鎖の先頭を代替するため)。ドミノをシグナルとして使う面では許容。
- ベルトも平地では DISPENSABLE になり得る(平地転がりの減衰が小さく素の転がりで届くことがある)。想定解はベルト使用のまま許容。
- §13.2-6 のタイミングスキャンは「ゲート側であるうごくゆか自身」はシフトしない(±32のシフト対象はボール到着時刻を決める他のギミックのみ)。
