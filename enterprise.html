株式会社ラクス | 事業戦略本部

Executive Summary

課題: 楽楽精算が1000名以上のエンタープライズ市場で勝つためには、機能不足ではなく、既存の巨大なエコシステム（SAP/Oracle等）との「接続性と共存」が最大の障壁となっている。

提案: 安易なSI（人月）や機能追加に走るのではなく、「Palantir型モデル（Ontology + FDE）」を採用する。

  • Technical: 既存ERPの上に「仮想統合レイヤー（Ontology）」を構築し、SaaS本体を汚さずに複雑な連携を吸収する。
  • Organizational: 導入エンジニア（FDE）を組織し、現場で作った接続ロジックを「資産（Archetype）」として製品に還流させる。

ラクスの現在地と「次の山」

SMB Market: Dominant

中堅・中小企業市場において、「楽楽精算」は圧倒的なシェアと認知を獲得。高収益なSaaSモデルが完成している。

Enterprise Market: The Challenge

売上1000億円を目指す上で不可欠な巨大市場。しかし、ここでの勝率はSMBほど高くない。

なぜか？ SMBの成功法則が通用しないからだ。

エンタープライズ顧客の「本当のペイン」

彼らが楽楽精算を導入できない（しない）理由は、「UIが使いにくいから」ではない。

既存エコシステムへの異物混入

SAP/Oracle/COMPANYといった堅牢な基幹システムが出来上がっており、そこに連携できないSaaSは「異物」でしかない。

データ整合性のリスク

「人事マスタはCOMPANYが正。楽楽精算とズレたら監査に通らない」。二重メンテは絶対に許容されない。

カオスな独自ルール

「A事業部の部長は、プロジェクトXの予算承認も兼ねる」といった、標準機能ではカバーしきれない組織力学。

Scenario 1: 人事マスタ連携 (The "Who")

VS: COMPANY / Workday / SuccessFactors

課題: 「Aさんは営業部の課長だが、プロジェクトXの責任者も兼務している」といった複雑な兼務・権限情報。

Ontologyの解:

  • 複数の人事DBから情報を仮想統合し、Ontology上で `Employee` オブジェクトを構築。
  • 「誰が承認すべきか」というロジックをOntologyで計算し、楽楽精算には「解決済みの承認者リスト」だけを渡す。

Logic Example

IF Request.Type == 'ProjectExpense':
  Approver = Ontology.Project(ID).Manager
ELSE:
  Approver = Ontology.Employee(ID).PrimaryDept.Manager

※このロジックを楽楽精算本体ではなく、Ontology層に持たせることで、本体の肥大化を防ぐ。

Scenario 2: 予算統制連携 (The "Control")

VS: SAP PS / Oracle Projects

課題: 経費を使ってから「予算オーバーでした」と判明する手戻り。

Ontologyの解: Advanced Validation

  • 申請ボタンが押された瞬間に、OntologyがSAPのAPIを叩きに行く。
  • 「予算残高不足なら警告を出し、申請をブロックする」という、リアルタイムな統制機能を提供。

Scenario 3: ERPへの書き戻し (The "Ledger")

VS: SAP S/4HANA / SuperStream-NX

課題: 仕訳データを送るだけの一方通行。ERP側での支払処理結果がわからない。

Ontologyの解: Write-back & Sync

  • 楽楽精算で承認完了 → ERPへ仕訳連携。
  • ERPで支払実行（FBデータ作成）。
  • OntologyがERPの「支払済」ステータスを検知し、楽楽精算側のステータスも「精算完了」に自動更新。

Two-way Sync

  • 楽楽精算: 仕訳データ送信
  • SAP: 伝票番号(DocNum)返却
  • SAP: 支払ステータス更新

組織戦略: FDE組織の創設

この戦略を実行するのは、既存のCSでもSEでもない、新しい職種である。

FDE (Forward Deployed Engineer)

「コードを書ける導入コンサルタント」。
顧客の現場最前線(Forward)に展開(Deploy)され、Ontologyを実装する。

Mission

顧客課題を解決しつつ、その解決策を「汎用資産（Archetype）」として製品に還流させること。

Skill Set

Python/TypeScriptの実装力 + 業務プロセス設計力 + 顧客折衝力。

FDE vs 客先常駐SE

客先常駐SE

  • ベクトル: 外向き (Client First)
  • 目的: 顧客の要望を100%満たす (御用聞き)。
  • 成果物: 顧客の資産 (個別コード)。
  • キャリア: 現場に埋没しがち。

FDE (Palantir Model)

  • ベクトル: 内向き (Product First)
  • 目的: 顧客課題を通じて製品を進化させる。
  • 成果物: 自社の資産 (Archetype)。
  • キャリア: エリートコース (Future PdM)。
